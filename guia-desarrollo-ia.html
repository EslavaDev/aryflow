Fundamentos

Los 4 Principios

El ciclo completo de desarrollo sigue cuatro principios fundamentales que garantizan calidad y eficiencia.

Checkpoints
1. Pensar primero
Brainstorming y especificacion antes de escribir una sola linea. SPEC.md + TODO.md compartidos por todos los agentes.
2. Recordar todo
Engram (explicito) + Claude-Mem (automatico). Las decisiones de enero estan disponibles en marzo.
3. Trabajar en paralelo
Tareas agrupadas en "waves" segun grafo de dependencias. Waves independientes corren en paralelo, cada agente en su propia copia aislada.
4. Revisar siempre
Checkpoint obligatorio despues de cada wave. Solo se avanza tras pasar la revision.

Conocimiento curado y estructurado con gestion de ciclo de vida. Solo se guarda lo que pasa criterios estrictos.

Escritura explicita con criterios
Solo se guarda: bug fixes con root cause, gotchas, decisiones arquitectonicas, convenciones establecidas.

04
Sistema de Memoria

Claude-Mem -- Memoria historica

Captura automatica cronologica + resumenes de sesion via HTTP API. Busqueda semantica con ChromaDB.

Captura automatica + resumenes HTTP
Captura en segundo plano. Los resumenes de sesion se envian via HTTP API al Stop hook. Sin lifecycle tags.
Busqueda semantica (ChromaDB)
"Que decidimos sobre autenticacion?" — busca por significado, no por clave exacta. Embeddings vectoriales.
Cronologica, sin deprecacion
Las entradas forman una linea de tiempo. No usan [ACTIVE]/[DEPRECATED] -- eso es exclusivo de engram.
Extraccion de descubrimientos
El Stop hook extrae descubrimientos de resumenes y los promueve a engram como entradas [ACTIVE] permanentes.

Siempre ambos, no fallback: Engram primero (conocimiento curado), Claude-Mem segundo (historia). Ambos se consultan siempre.

06
Gestion de Conocimiento

Proteccion contra Conocimiento Obsoleto

Sin una estrategia de gestion, las entradas contradictorias se apilan. Engram usa tres capas de proteccion (claude-mem es cronologico y no necesita gestion de obsolescencia).

Capa 1: Versionado con estado (engram)
Cada entrada en engram incluye [ACTIVE] o [DEPRECATED] con referencia a la que la reemplaza. Trazabilidad completa.
[ACTIVE] [DEPRECATED]
Capa 2: Regla de recencia
Cuando hay entradas en conflicto, la entrada [ACTIVE] mas reciente gana. Los agentes ignoran todas las entradas [DEPRECATED].
Automatica
Capa 3: Limpieza periodica
Al final de cada hito: buscar [DEPRECATED], verificar reemplazo, eliminar permanentemente lo obsoleto.
Por hito
"El conocimiento no expira por tiempo. Expira por reemplazo."
Una decision de hace 6 meses puede seguir vigente. No se borra por antiguedad, solo cuando algo la reemplaza.

Limpieza periodica: Al final de cada hito mayor, el agente knowledge-gc busca entradas [DEPRECATED], verifica que la entrada que las reemplazo sigue [ACTIVE], y elimina las obsoletas permanentemente.

Nota: los marcadores se escriben siempre en ingles ([ACTIVE]/[DEPRECATED]) para consistencia entre agentes.

11
Ejemplo Practico

Retomar trabajo despues de una interrupcion

El sistema detecta progreso previo y reanuda exactamente donde se quedo. Sin repetir trabajo.

Sesion anterior: se interrumpio en Wave 3 de 5

> /execute-spec 001-agent-registry

→ Detecta progreso previo en engram + claude-mem
→ "Reanudando desde Wave 3. Waves 1-2 completadas."
→ Continua desde Wave 3 sin repetir trabajo
→ Todo el conocimiento de Waves 1-2 disponible
Progreso persistente
Despues de cada wave, el estado se guarda en engram y la sesion en claude-mem. Si la sesion muere, el progreso sobrevive.
Conocimiento acumulado
Los descubrimientos de waves anteriores estan disponibles. No se pierde contexto entre sesiones.

Clave: La capacidad de reanudar es lo que hace viable trabajar en features grandes que exceden una sola sesion de IA.

13
Aislamiento

Branches y Worktrees

Branch es el modo por defecto. Worktree se usa para aislamiento por wave cuando hay ejecucion paralela.

Branch (por defecto)
Modo estandar. Un branch de desarrollo donde se trabaja secuencialmente. Simple, directo, suficiente para la mayoria de specs.
Git Worktree (aislamiento por wave)
Cada wave recibe su propio worktree aislado. Los agentes dentro de una wave trabajan sobre esa copia. Al completar, se integra via merge agent.
Branch principal
Orquestador
→
WT Wave 1
3 agentes
WT Wave 2
2 agentes
→
Merge
merge-wave agent

Waves en paralelo: Si Wave 3 depende de Wave 1 (no de Wave 2), Wave 2 y Wave 3 corren en paralelo. El grafo de dependencias determina que puede ejecutarse simultaneamente.

Agente de merge dedicado: Despues de cada wave, un agente especializado integra los worktrees. Lee el spec, resuelve conflictos automaticamente, instala dependencias una sola vez, y solo escala al desarrollador si el conflicto es ambiguo.
